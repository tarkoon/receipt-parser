## 1. Overview

receipt-scanner is the most popular project in this research batch (98 stars, 32 forks) and the most technically ambitious. It is a fully custom ML-based OCR system built from scratch, implementing both CNN and CRNN (Convolutional Recurrent Neural Network) architectures for character-level text recognition on receipts. Notably, it does **not use any cloud OCR service** -- it trains its own OCR models using synthetic receipt data generated by a built-in receipt generator.

The project represents an academic deep-dive into the OCR problem itself, not just the extraction/parsing problem. It includes a synthetic data generation pipeline, two neural network architectures, line/word segmentation algorithms, and spatial text reconstruction.

## 2. Architecture & How It Works

### Overall Pipeline

```
Receipt Image
  -> ReceiptProcessor: Line Segmentation (cut_lines)
     -> Horizontal projection profiling to find text line boundaries
  -> ReceiptProcessor: Word/Letter Segmentation (cut_letters via bounding boxes)
  -> CRNNModel: Character Recognition
     -> CNN feature extraction (VGG-style)
     -> Bidirectional LSTM sequence labeling
     -> CTC beam search decoding
  -> output_text: Spatial Reconstruction
     -> Cluster by vertical position (line grouping)
     -> Sort left-to-right within lines
     -> Concatenate into readable text
```

### CRNN Architecture (ShadowNet)

The CRNN model is a three-stage neural network:

**Stage 1 -- Feature Extraction (CNN):**
- VGG-style convolutional backbone
- Conv layers: 64 -> 128 -> 256 -> 512 channels
- Batch normalization at deeper layers
- Asymmetric pooling: reduces height while preserving width
- Output: `[batch, 1, 25, 512]` feature maps

**Stage 2 -- Map-to-Sequence:**
- Squeezes height dimension: `[batch, 25, 512]`
- Each position in the width dimension becomes a timestep

**Stage 3 -- Sequence Labeling (RNN):**
- Stacked bidirectional LSTM layers
- Forward + backward hidden states
- Dropout (keep_prob=0.5) during training
- Fully connected projection to class logits
- CTC beam search decoder for final text output
- 71 character classes (letters, digits, punctuation)

### Synthetic Receipt Generator

The `ReceiptGenerator` is particularly sophisticated:

1. **Configuration:** Random layout parameters -- separators, columns, margins, pricing ranges, visual distortions
2. **Content Generation:** Random items with names, prices, quantities, and totals via `items_creator.py`
3. **Rendering:** Text drawn on paper texture backgrounds with configurable fonts, margins, and Gaussian blur for realism
4. **Character Extraction:** Individual character bounding boxes extracted via `letter_cutter.py` for training labels
5. **Data Augmentation:** Augmentor library for additional distortions (rotation, skew, noise)
6. **Output:** Image + text + character-level bounding boxes + configuration JSON per sample

### Text Reconstruction

The `output_text.py` module reconstructs readable text from OCR detections:

1. **Vertical clustering:** Groups text by y-coordinate, using `maxgap = avg_height * 0.4` as the line-break threshold
2. **Horizontal sorting:** Within each cluster, sorts text by x-coordinate (left-to-right)
3. **Concatenation:** Joins words with spaces, lines with newlines

## 3. Key Features

- **Fully custom OCR** -- no dependency on Google Cloud Vision, Tesseract, or any external service
- **Two model architectures** -- CNN (simpler) and CRNN (primary, more accurate) for comparison
- **Synthetic data generation** -- creates unlimited training data with configurable receipt layouts, fonts, and distortions
- **Character-level bounding boxes** -- training labels include per-character spatial annotations
- **CTC decoding** -- handles variable-length text sequences without explicit segmentation
- **Paper texture simulation** -- synthetic receipts use real paper texture backgrounds for domain realism
- **Font diversity** -- 40+ monospace fonts for training data variation
- **Academic rigor** -- includes a project report PDF documenting methodology, experiments, and results
- **Pre-trained weights** -- CRNN checkpoint files included in the repo for immediate inference
- **Spatial text reconstruction** -- reassembles detected text into spatially correct reading order

## 4. Japanese Support

**None, and fundamentally limited.** The system has several hard barriers to Japanese support:

- **Character dictionary:** Only 71 character classes (ASCII letters, digits, basic punctuation). Japanese would need 2000+ kanji minimum plus kana.
- **Monospace font assumption:** All 40+ training fonts are Latin monospace. Japanese characters are typically proportional or half/full-width mixed.
- **Fixed sequence length:** `seq_length=25` is designed for short Latin text segments. Japanese text segments have different length distributions.
- **Training data:** Synthetic receipt generator creates English-only receipts with Western pricing formats.
- **No CJK awareness:** Line segmentation, letter cutting, and text reconstruction assume Latin character widths and spacing.

To support Japanese, one would need to: expand the character dictionary to 3000+ classes, add Japanese fonts, retrain from scratch with Japanese text, and modify the segmentation algorithms for CJK character widths. This is essentially a complete rewrite.

## 5. Strengths vs Our Project

- **No cloud dependency:** The entire OCR pipeline runs locally. No Google Cloud Vision costs, no network latency, no API rate limits, no privacy concerns about sending data to cloud services.
- **Full control over OCR model:** They can fine-tune the OCR model on specific receipt types, fonts, or degradation patterns. We are limited to whatever Google Cloud Vision provides.
- **Synthetic data generation:** The receipt generator creates unlimited, perfectly-labeled training data. This solves the common ML problem of insufficient training data for document understanding. The approach of using paper textures, font variety, and augmentation is well-engineered.
- **Two-model comparison:** Having both CNN and CRNN allows architectural comparison. Their CRNN approach (shared with industry-standard text recognition systems like Tesseract 5's LSTM mode) demonstrates that competitive OCR can be built from scratch.
- **Academic documentation:** The project report provides detailed methodology. Our project lacks formal documentation of our architectural decisions and their justifications.
- **Spatial text reconstruction:** Their vertical clustering + horizontal sorting algorithm for reassembling OCR output into reading order is clean and could be useful as a preprocessing step.

---

## 6. Weaknesses vs Our Project

- **OCR-only, no extraction:** This project only does character recognition. It does not extract structured fields (dates, totals, merchant names, line items). Our pipeline goes from image to fully structured JSON.
- **English/Latin only:** 71 character classes cannot handle Japanese. Our Google Cloud Vision integration handles any language.
- **TensorFlow 1.x era:** Uses old TensorFlow APIs (tf.Session, tf.train.Saver, tf.nn.ctc_beam_search_decoder). This is circa-2018 TensorFlow, now deprecated.
- **No extraction/parsing layer:** After OCR, the output is just reconstructed text. No LLM, no schema, no validation. It solves a different (smaller) problem than we do.
- **Accuracy limitations:** A custom-trained CRNN with 71 classes on synthetic data will not match Google Cloud Vision's accuracy, which is trained on billions of real-world samples.
- **No confidence scoring:** The CTC decoder outputs text but no per-character or per-word confidence scores. Our Google Cloud Vision integration provides confidence for each detected word.
- **No receipt-specific post-processing:** No price formatting, no date parsing, no total verification. Just raw text output.
- **Synthetic-real domain gap:** Models trained on synthetic data often struggle with real-world receipt degradation (creases, fading, angle, partial occlusion). Google Cloud Vision is trained on real-world data at massive scale.
- **Fixed input size:** The model expects specific image dimensions. Our pipeline handles arbitrary image sizes via the Cloud Vision API.
- **Maintenance burden:** Running your own OCR model means maintaining training infrastructure, handling model updates, and debugging model-specific issues.

## 7. What We Can Learn

1. **Synthetic data generation for testing:** Their receipt generator creates realistic synthetic receipts with known ground truth. We could build a similar generator for Japanese receipts to augment our test fixtures -- generating receipts with specific challenging characteristics (faded text, rotated images, overlapping text, unusual fonts) to stress-test our pipeline.

2. **Spatial text reconstruction algorithm:** Their vertical clustering + horizontal sorting approach for reassembling OCR text into reading order is simple and effective. We could use a similar algorithm as a sanity check on Google Cloud Vision's text ordering, or as a fallback when the OCR returns text in unexpected order.

3. **Paper texture and augmentation for realism:** Their use of real paper textures, Gaussian blur, and Augmentor-based distortions creates realistic training data. If we ever fine-tune a model, these techniques would be valuable.

4. **Project report as documentation:** Their academic report documenting methodology, experiments, and results is a good practice. We could create a similar technical document explaining our confidence routing system, subset-sum tax matching, and other novel techniques.

5. **Local OCR as a privacy option:** For users who cannot send receipt images to cloud services (privacy regulations, corporate policies), a local OCR option (even if less accurate) could be valuable. This project proves it is feasible, though we would use a modern model (PaddleOCR, EasyOCR, or Tesseract 5) rather than this specific CRNN.

## 8. Recommendation

**Do not adopt this tool.** It solves a fundamentally different problem (OCR character recognition) than our project (receipt information extraction). It is also English-only, built on deprecated TensorFlow 1.x, and cannot match cloud OCR accuracy. However, two ideas are worth pursuing:

- **Synthetic receipt generator:** Build a Japanese receipt image generator for expanding our test fixtures. Use paper textures, multiple fonts, and augmentation to create realistic test images with known ground truth. This would let us test edge cases (faded text, unusual layouts, rare merchants) without needing real receipt images.
- **Local OCR fallback:** Consider adding PaddleOCR or EasyOCR as an optional local OCR backend for users who cannot use cloud services, following this project's pattern of local inference. This would not replace Google Cloud Vision as the primary engine but would provide a privacy-preserving alternative.
